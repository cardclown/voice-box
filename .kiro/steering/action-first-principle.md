# 行动优先原则

## 核心原则

**首要任务是解决问题，而不是写报告**

禁止在问题未解决的情况下，创建各种"最终报告"、"完成总结"、"问题汇总"等文档来逃避问题。

---

## ⚠️ 严格禁止的行为

### 1. 问题未解决就写"完成报告"

#### ❌ 错误做法

```bash
# 遇到编译错误
mvn clean install
# [ERROR] 编译失败

# 错误：立即创建"问题总结"
cat > COMPILATION_ISSUE_SUMMARY.md << 'EOF'
# 编译问题总结

## 问题描述
编译失败

## 可能原因
1. 依赖问题
2. 代码错误
3. 环境问题

## 建议方案
1. 检查依赖
2. 修复代码
3. 更新环境

## 下一步
待进一步分析
EOF

# 然后就不管了！这是逃避问题！
```

#### ✅ 正确做法

```bash
# 遇到编译错误
mvn clean install
# [ERROR] 编译失败

# 正确：立即分析和解决
mvn clean install -X  # 查看详细错误

# 分析错误原因
# - 是哪个文件的错误？
# - 具体是什么错误？
# - 如何修复？

# 修复问题
vim src/main/java/...  # 修复代码

# 验证修复
mvn clean install

# 确认成功后，简单记录（可选）
echo "修复编译错误：[具体问题]" >> docs/CHANGELOG.md
```

### 2. 创建大量"FINAL"、"COMPLETE"文档

#### ❌ 禁止的文档命名

- `FINAL_COMPLETION_REPORT.md` - 问题还没解决就说"最终完成"
- `COMPLETE_IMPLEMENTATION_SUMMARY.md` - 实现还有问题就说"完整实现"
- `DEPLOYMENT_SUCCESS_FINAL.md` - 部署还失败就说"成功"
- `PROBLEM_RESOLUTION_SUMMARY.md` - 问题还在就说"已解决"
- `NEXT_STEPS.md` - 当前步骤都没完成就规划下一步

#### ✅ 允许的文档（在真正完成后）

- `CHANGELOG.md` - 记录实际完成的变更
- `README.md` - 功能说明（功能确实可用）
- `TROUBLESHOOTING.md` - 真实遇到并解决的问题

### 3. 用文档掩盖问题

#### ❌ 错误示例

```markdown
# 部署问题总结

## 当前状态
部署遇到一些问题

## 问题列表
1. 编译失败 - 待解决
2. 依赖冲突 - 待解决
3. 配置错误 - 待解决

## 建议
建议进一步调查

## 结论
需要更多时间分析
```

**问题**：写了一堆废话，但一个问题都没解决！

#### ✅ 正确做法

不要写文档，直接解决问题：

```bash
# 1. 解决编译失败
mvn clean install -X
# 查看错误，修复代码
vim src/main/java/...
mvn clean install  # 验证修复

# 2. 解决依赖冲突
mvn dependency:tree
# 分析冲突，修改 pom.xml
mvn clean install  # 验证修复

# 3. 解决配置错误
cat config.properties
# 修正配置
vim config.properties
./restart-all.sh  # 验证修复

# 全部解决后，简单记录
git commit -m "修复部署问题：编译错误、依赖冲突、配置错误"
```

## 正确的工作流程

### 遇到问题时

```
发现问题
    ↓
❌ 不要：创建"问题总结"文档
✅ 应该：立即分析和解决
    ↓
分析问题原因
    ↓
尝试解决方案
    ↓
验证是否解决
    ↓
问题解决了？
    ├─ 是 → 继续工作（可选：简单记录）
    └─ 否 → 继续尝试其他方案
```

### 完成任务时

```
任务完成
    ↓
验证功能正常
    ↓
测试通过
    ↓
提交代码
    ↓
✅ 可以：简单记录完成的内容
❌ 不要：创建冗长的"完成报告"
```

## 文档创建规则

### 何时可以创建文档

1. **功能确实完成** - 代码工作正常，测试通过
2. **问题确实解决** - 不再有错误，功能正常
3. **有实际价值** - 文档对未来有参考价值

### 何时不应该创建文档

1. **问题还没解决** - 不要写"问题总结"逃避
2. **功能还不能用** - 不要写"完成报告"自欺欺人
3. **只是记录失败** - 失败了就继续尝试，不要记录失败

### 文档内容规则

#### ❌ 禁止的内容

```markdown
## 问题
遇到了一些问题

## 原因
可能是 A、B、C

## 建议
建议进一步分析

## 下一步
待确定
```

**问题**：全是废话，没有任何实际行动！

#### ✅ 允许的内容

```markdown
## 修复的问题
1. 编译错误：缺少依赖 X，已添加到 pom.xml
2. 配置错误：端口冲突，已修改为 8081
3. 测试失败：空指针异常，已添加空值检查

## 验证结果
- 编译成功
- 服务启动正常
- 所有测试通过
```

**特点**：具体、明确、已完成！

## 常见逃避行为识别

### 行为 1: 创建多个"总结"文档

```bash
ls .kiro/specs/voice-interaction/
COMPILATION_ISSUE_SUMMARY.md
DEPLOYMENT_ISSUE_SUMMARY.md
PROBLEM_RESOLUTION_SUMMARY.md
FINAL_COMPLETION_REPORT.md
NEXT_STEPS.md
CURRENT_STATUS.md
...
```

**问题**：文档一大堆，问题一个没解决！

**正确做法**：删除这些文档，直接解决问题。

### 行为 2: 反复创建"下一步"文档

```markdown
# NEXT_STEPS.md

## 下一步计划
1. 分析问题
2. 尝试解决
3. 验证结果

## 预期时间
待定
```

**问题**：规划了一堆，但一步都没执行！

**正确做法**：不要规划，直接执行第一步。

### 行为 3: 创建"当前状态"文档

```markdown
# CURRENT_STATUS.md

## 当前进度
50%

## 遇到的问题
- 问题 1
- 问题 2
- 问题 3

## 待办事项
- 解决问题 1
- 解决问题 2
- 解决问题 3
```

**问题**：记录了一堆待办，但一个都没做！

**正确做法**：不要记录待办，直接去做。

## 正确的问题解决流程

### 示例：编译失败

#### ❌ 错误流程

```bash
# 1. 编译失败
mvn clean install
# [ERROR]

# 2. 创建问题文档
cat > COMPILATION_ISSUE.md << 'EOF'
编译失败，需要分析原因
EOF

# 3. 创建解决方案文档
cat > SOLUTION_PLAN.md << 'EOF'
计划分析依赖、检查代码、更新环境
EOF

# 4. 创建进度文档
cat > PROGRESS.md << 'EOF'
当前进度：分析中
EOF

# 5. 然后就不管了
```

**结果**：创建了 3 个文档，问题还在！

#### ✅ 正确流程

```bash
# 1. 编译失败
mvn clean install
# [ERROR] 

# 2. 查看详细错误
mvn clean install -X
# 发现：缺少依赖 com.example:some-lib

# 3. 添加依赖
vim pom.xml
# 添加依赖

# 4. 重新编译
mvn clean install
# [INFO] BUILD SUCCESS

# 5. 验证
mvn test
# [INFO] Tests run: 10, Failures: 0

# 6. 提交
git add pom.xml
git commit -m "添加缺失的依赖 some-lib"

# 完成！不需要任何"总结"文档
```

## 文档清理规则

### 定期清理无用文档

```bash
# 删除这些逃避问题的文档
rm -f *_ISSUE_SUMMARY.md
rm -f *_PROBLEM_*.md
rm -f NEXT_STEPS.md
rm -f CURRENT_STATUS.md
rm -f *_FINAL_*.md
rm -f *_COMPLETE_*.md

# 只保留真正有价值的文档
# - README.md
# - CHANGELOG.md
# - 实际的设计文档
# - 实际的使用指南
```

### 文档审查清单

创建文档前问自己：

- [ ] 问题真的解决了吗？
- [ ] 功能真的完成了吗？
- [ ] 这个文档有实际价值吗？
- [ ] 我是在逃避问题吗？
- [ ] 我应该先解决问题而不是写文档吗？

如果有任何一个答案是"否"或"不确定"，**不要创建文档，去解决问题**。

## 时间分配原则

### 正确的时间分配

- **90%** - 解决问题、写代码、测试
- **10%** - 必要的文档（README、注释等）

### 错误的时间分配

- **50%** - 写各种"总结"、"报告"、"计划"
- **30%** - 创建"问题分析"、"解决方案"文档
- **20%** - 实际解决问题

**结果**：问题没解决，文档一大堆！

## 团队协作规范

### 汇报工作时

#### ❌ 错误汇报

"我创建了问题分析文档、解决方案文档、进度报告文档..."

**问题**：说了一堆文档，问题解决了吗？

#### ✅ 正确汇报

"我修复了编译错误、解决了依赖冲突、配置已更新，服务正常运行。"

**特点**：说的都是实际完成的工作！

### 请求帮助时

#### ❌ 错误请求

"我写了一个问题总结文档，请看一下。"

**问题**：让别人看文档有什么用？

#### ✅ 正确请求

"我遇到编译错误：[具体错误信息]，已尝试 [方案 A、B]，仍未解决，能帮我看看吗？"

**特点**：提供具体信息，说明已尝试的方案！

## 自我检查

### 每天结束时问自己

1. **今天解决了几个问题？** - 具体的问题
2. **今天完成了什么功能？** - 可以运行的功能
3. **今天写了几个文档？** - 如果超过 2 个，可能有问题

### 警告信号

如果你发现自己：

- ✅ 创建了超过 3 个"总结"类文档
- ✅ 文档名包含"FINAL"、"COMPLETE"但功能还不能用
- ✅ 花在写文档的时间超过写代码
- ✅ 问题还在但已经写了"解决方案"文档

**说明**：你在逃避问题！停止写文档，去解决问题！

---

## 记住

1. **代码 > 文档** - 能用的代码比漂亮的文档重要
2. **行动 > 计划** - 实际解决问题比规划重要
3. **结果 > 过程** - 问题解决了比记录过程重要
4. **简洁 > 冗长** - 简单记录比长篇大论重要

## 口号

**少说多做，问题优先！**

- 遇到问题？解决它！
- 功能不能用？修复它！
- 测试失败？调试它！
- 不要写文档，去写代码！

## 例外情况

### 唯一允许详细文档的情况

1. **复杂的架构设计** - 需要团队讨论和评审
2. **API 文档** - 供其他开发者使用
3. **部署指南** - 供运维人员参考
4. **故障排查指南** - 记录真实遇到并解决的问题

### 但即使这些文档也要

- 简洁明了
- 实际可用
- 及时更新
- 不要冗余

如果答案是后者，**立即停止，去解决问题！**
